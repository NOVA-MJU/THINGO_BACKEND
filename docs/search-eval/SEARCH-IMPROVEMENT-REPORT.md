# 명지대 통합검색 개선 보고서 (쉽게 읽는 버전)

> 이 문서는 "명지대 통합검색이 왜 느리고 결과가 안 나왔는지, 그걸 어떻게 찾아내고 고쳐서
> 어떤 성과를 냈는지"를 **누구나(전공 몰라도) 이해할 수 있게** 풀어 쓴 보고서입니다.
> 대상 API: `GET /api/v2/search/detail` (PostgreSQL 기반 통합검색)
> 곁들임 문서: 숫자 위주 평가는 [eval-report.md](eval-report.md), 질의별 원시결과는 [eval-results.md](eval-results.md),
> 원래 설계 요구사항은 [../search-spec.md](../search-spec.md).

## 0. 30초 요약

검색창에 단어를 치면 **4초**나 걸렸고, 5번 중 1번은 **"결과 없음"**이 떴습니다.
원인을 측정으로 찾아내 고친 결과:

| 무엇 | 고치기 전 | 고친 후 | 한마디 |
|------|-----------|---------|--------|
| 검색 속도(P95) | 4,201ms (4초) | 약 230ms (0.2초) | 약 18배 빨라짐 |
| 결과 없음 비율 | 21% (15/72) | 3% (2/72) | 거의 항상 결과가 나옴 |
| 의미 없는 입력(예: "ㅁㅈㄷ") | 약 3,900ms | 약 20ms | 헛고생 안 하고 즉시 응답 |

개선의 핵심은 두 가지입니다. 첫째, 예전에는 검색할 때마다 전체 문서를 하나씩 훑어서 느렸는데,
색인을 제대로 활용하도록 바꿔 즉시 찾게 했습니다. 둘째, 사용자의 검색어를 데이터 저장 방식과
똑같이 처리하도록 바꿔 "결과 없음"이 뜨던 경우가 거의 사라졌습니다.

이후 품질 보강으로, 같은 키워드라면 **지금 유효한 공지가 먼저** 뜨도록 마감일 기반 재랭킹을
더했습니다(마감 지난 공지는 후순위, 학칙처럼 기한 없는 문서는 우대). 자세한 설계는
[RANKING-VALIDITY-DESIGN.md](RANKING-VALIDITY-DESIGN.md) 참고.

## 1. 검색은 원래 어떻게 동작하나 (기본기)

명지대 통합검색은 8종(공지·학사일정·학과공지·총학공지·학과일정·커뮤니티·명대뉴스·방송)을
한 표(`unified_search_index`)에 모아두고 찾습니다. 표의 각 줄(문서)에는:

- **title / content** : 원문 제목·본문
- **search_tokens** : 제목·본문을 한국어 형태소 분석기 **Komoran**으로 잘게 쪼갠 "검색용 단어들"
- **search_vector** : 데이터베이스 전문검색(FTS)이 쓰는 색인 형태

아래 네 가지 용어만 편하게 이해하면, 이 보고서의 나머지는 자연스럽게 읽힙니다. 각 용어가
"무엇인지"와 "왜 검색에 필요한지"를 함께 설명하겠습니다.

**형태소 분석 (Komoran)**
한국어는 단어를 붙여 쓰고 조사가 따라붙기 때문에, 글자를 그대로 비교하면 검색이 잘 맞지 않습니다.
예를 들어 "수강신청했어요"라는 문장은 사람 눈에는 '수강신청'이 보이지만, 컴퓨터에게는 그냥 긴 글자
덩어리일 뿐입니다. 그래서 Komoran이라는 한국어 형태소 분석기로 이 문장을 `수강`, `신청` 같은
의미 단위로 미리 쪼개 둡니다. 이렇게 쪼갠 단어들을 저장해 두면, 사용자가 어떻게 입력하든 의미가
같은 글을 찾아낼 수 있습니다.

**전문검색(FTS)과 색인(GIN)**
FTS는 "이 단어가 들어 있는 문서가 어디인가"를 찾아 주는 검색 방식이고, 그 속도의 비결이 색인(GIN)
입니다. 색인은 "어떤 단어가 어떤 문서에 들어 있는지"를 미리 정리해 둔 표입니다. 덕분에 검색할 때
모든 문서의 본문을 처음부터 읽을 필요 없이, 색인만 보고 해당 문서로 곧장 갈 수 있어, 단어 검색은
문서 수가 많아도 매우 빠릅니다.

**tsvector**
FTS가 사용하는 색인의 실제 형태입니다. 위에서 쪼갠 단어들을 데이터베이스가 검색하기 좋은 모양으로
정리해 둔 것이라고 생각하면 됩니다.

**trigram (글자 3개 비교)**
trigram은 단어를 "글자 3개씩 겹쳐 가며" 잘라 만든 조각으로 두 단어가 얼마나 닮았는지를 재는
방법입니다. 예를 들어 '장학금'은 ` 장`, `장학`, `학금`, `금 ` 같은 조각으로 나뉩니다. '장학금'과
오타인 '장학근'은 이 조각들이 거의 겹치기 때문에 "많이 닮았다(유사도가 높다)"고 판단할 수 있어,
오타에 강합니다. 다만 이 방법은 **어떻게 쓰느냐에 따라 빠르기도 하고 매우 느리기도 한데**, 바로
이 지점이 옛 검색이 느렸던 핵심 원인이었습니다(자세한 내용은 2-A장에서 설명합니다).

## 2. 무엇이 문제였나 (Problem)

문제를 "느낌"이 아니라 "숫자"로 확인하려고, 실제 학생이 검색할 법한 질문 **72개**를 만들어
서버에 던지고 결과를 모았습니다(데이터셋 구성은 4장 참고). 세 가지 큰 문제가 보였습니다.

### 문제 1. 검색이 무조건 4초 (제일 심각)

- 무엇을 검색하든, 결과가 1건이든 1000건이든 **항상 약 4초**. (빈 검색만 0.08초)
- 원인: 검색 조건에 `similarity(검색용단어들, 검색어) > 0.1`(trigram 유사도)이 들어 있었습니다.
  이 방식은 **색인을 못 씁니다.** 그래서 데이터베이스가 **전체 문서(3,975건)를 한 줄씩** 꺼내
  글자 비교를 했습니다. 색인을 활용하지 못하고 전체 문서를 처음부터 훑은 셈입니다.

### 문제 2. 5번 중 1번은 "결과 없음" (recall 21%)

- "출결 규정", "도서관 좌석", "학식 메뉴", "성적우수장학금" 같은 평범한 검색이 **0건**.
- 원인 두 가지:
  - **검색어를 안 쪼갰다**: 저장은 형태소로 쪼개 했는데(`수강`,`신청`), 검색은 사용자가 친 그대로
    비교했습니다. 그래서 "성적우수장학금"처럼 붙여 친 말이 쪼개진 색인과 안 맞아 0건.
  - **여러 단어는 전부 'AND'**: "출결 규정"은 `출결 그리고 규정`으로 처리돼, 두 단어가 **동시에**
    든 문서가 없으면 0건. "졸업하려면 학점 얼마나 들어?" 같은 문장은 '얼마나','어떻게' 같은
    의미 없는 말까지 AND로 묶여 0건이 됐습니다.

### 문제 3. 엉뚱한 문서가 위에 (precision)

- "국가장학금"을 치면 상위에 "명대신문사", "The…" 같은 무관한 게 떴습니다.
- 원인: 색인이 **제목과 본문을 똑같이 취급**해서, 본문에 단어가 잠깐 스친 글이 제목에 딱 맞는 글을
  눌러버렸습니다.

## 2-A. 왜 "4초"나 걸렸을까? (속도 문제 깊이 보기)

가장 중요한 부분이라 따로 떼어 자세히 설명합니다.

### 핵심: "색인으로 답할 수 있는 질문" vs "없는 질문"

검색 조건이 이렇게 돼 있었습니다.

```
WHERE search_vector @@ ...                              (A) 단어 포함?
   OR similarity(search_tokens, 검색어) > 0.1           (B) 70%처럼 '비슷한 정도'가 0.1 넘나?
   OR title ILIKE '%검색어%'                            (C) 제목에 부분문자열?
```

- **(A) "이 단어가 든 문서?"** → 색인(GIN)에 정리돼 있어 **즉시 답**합니다.
- **(B) "유사도가 0.1보다 큰 문서?"** → 색인이 **답할 수 없는 질문**입니다.
  색인은 "단어 → 문서 목록"만 알지, "각 문서가 검색어와 몇 % 비슷한지"는 미리 안 적어둡니다.
  그래서 데이터베이스는 어쩔 수 없이 **모든 문서(3,975건)를 한 줄씩 꺼내**, 각 문서의
  `search_tokens`(제목과 본문을 모두 쪼개 이어 붙인 **수 KB 길이의 긴 글자열**)와 검색어를
  **글자 3개 단위로 매번 새로 비교 계산**했습니다. 즉 (A)는 색인에 이미 정리된 정보로 즉시 답하지만,
  (B)는 미리 정리해 둔 정보가 없어 문서마다 처음부터 계산해야 합니다.

여기서 중요한 사실이 하나 있습니다. **trigram 자체는 색인을 쓸 수 있습니다.** 단, `단어 % 단어`라는
전용 연산자(`%`) 형태로 썼을 때만, 미리 만들어 둔 trigram 색인(어떤 글자 조각이 어떤 문서에 있는지)을
활용합니다. 그런데 옛 코드는 `similarity(a, b) > 0.1`처럼 **유사도를 함수로 직접 계산해 그 값을 숫자와
비교**하는 형태였습니다. 이 형태에서는 데이터베이스가 색인을 끼워 넣을 방법이 없어, 모든 문서의
유사도를 하나씩 계산할 수밖에 없습니다. 같은 trigram 기술인데도 "연산자로 쓰면 색인 사용, 함수로 쓰면
전체 계산"이라는 차이가 있었고, 옛 코드는 후자였던 것입니다.

이처럼 함수 결과를 비교하는 조건은 색인을 활용하지 못해(전문 용어로 **비-sargable**) 결국
**전체 훑기(Sequential Scan)**가 됩니다. 이것이 "결과 건수와 상관없이 항상 느렸던" 근본 이유입니다.

### 느림을 키운 추가 요인

1. **같은 계산을 두 번**: `similarity(...)`가 "거르는 조건(WHERE)"에도, "점수 매기기"에도 들어 있어,
   걸러진 문서마다 무거운 글자 비교를 한 번 더 했습니다.
2. **긴 본문 하이라이트**: 결과로 보여줄 글의 본문에서 검색어 위치를 잘라내는 작업(ts_headline)도
   본문이 길면 비용이 큽니다.
3. **원격 DB**: 운영 DB가 원격(AWS)이라, 무거운 쿼리일수록 그 비용이 더 도드라졌습니다.

### "그게 진짜 범인"이라는 증거

- **대조군**: 검색어가 없는 "최신순 목록"은 (B) 계산이 없어 **0.08초**로 빨랐습니다.
  같은 DB·같은 네트워크인데 키워드 검색만 4초 → 느림의 원인은 네트워크가 아니라 **(B) 계산**.
- **단계적 실측**: trigram을 점수에서 빼니 일부 빨라졌고(2~4초), **검색 조건(WHERE)에서도 완전히
  빼니 0.2초**로 떨어졌습니다. 즉, "검색 조건의 trigram 전체 훑기"가 결정적 범인임을 숫자로 확인.

### 그래서 한 일

(B)·(C)를 빼고 **색인으로 답할 수 있는 (A)만** 남겼습니다(쪼갠 검색어로). 오타 대응은 trigram 대신
"검색어를 형태소로 쪼개 부분 매칭"으로 대체했고, 의미 없는 입력은 아예 DB를 안 치게 했습니다.
→ 전체 훑기가 사라지면서 4초가 0.2초가 됐습니다.

## 3. 어떻게 접근했나 (Approach)

추측으로 손대지 않고 **"측정 → 원인 → 수정 → 다시 측정"**을 반복했습니다.

```
1) 평가 도구 제작      질문 72개(query-bank.json) + 자동 실행 스크립트
2) 기준선 측정         "지금" 상태 기록: 4초, 21% 결과없음 (이게 비교 기준)
3) 원인 분석           코드를 읽어 느림/0건의 진짜 원인 특정
4) 한 번에 하나씩 수정  고칠 때마다 같은 72개 재실행 → 좋아졌는지 숫자로 확인
5) 회귀 잡기           수정하다 "전과" 검색이 깨진 것도 즉시 발견·복구
```

이렇게 자동 측정을 깔아둔 덕분에, "고친 것 같다"가 아니라 **"몇 ms로, 몇 % 좋아졌다"**를
매 단계 증명할 수 있었습니다.

> 참고: 운영 DB를 직접 들여다보는 건(EXPLAIN) 권한상 막혀 있어서, 원인은 **코드 분석**으로
> 확정했습니다. 대신 결과는 실제 서버 엔드포인트로 매번 측정했습니다.

## 4. 테스트 데이터셋은 어떻게 구성했나 (중요)

검색 품질은 "내가 검색해보니 잘 되네"로 판단하면 안 됩니다. **다양한 사람의 다양한 표현**으로
재봐야 진짜 약점이 보입니다. 그래서 두 축으로 72개 질문을 설계했습니다.

### 축 A — "무엇을 궁금해하나" (의도 8종)

명지대 학생이 학교에 대해 검색할 영역 전체를 덮도록 8개로 나눴습니다.

| 의도 | 예시 질문 | 왜 넣었나 |
|------|-----------|-----------|
| 학사/수강 | 수강신청, 졸업요건, 휴학, 계절학기, 전과 | 가장 검색량 많은 핵심 |
| 장학 | 국가장학금, 교내장학금, 장학금 신청 기간 | 금전 관련, 자주 찾음 |
| 취업/진로 | 현장실습, 채용설명회, 공모전, 인턴십 | 고학년 수요 |
| 학칙/규정 | 학사경고, 출결 규정, 등록금 환불, 제적 | "규정이 궁금" 케이스 |
| 시설/생활 | 도서관 좌석, 기숙사 신청, 셔틀버스, 학식 | 일상 생활 정보 |
| 학생활동 | 동아리 모집, 총학생회, 축제, 학생회비 | 교내 활동 |
| 행정 | 증명서 발급, 학생증 재발급, 전자출결 | 서류·시스템 |
| 뉴스/방송 | 명대신문, 명지대 방송국 | 외부 콘텐츠 도메인 |

→ 특정 분야만 잘 되고 다른 분야는 안 되는 "편식"을 잡기 위함.

### 축 B — "어떻게 입력하나" (입력형 9종)

**같은 의도라도 사람마다 다르게 칩니다.** 각 입력형은 특정 약점을 일부러 건드립니다.

| 입력형 | 예시 | 무엇을 시험하나 |
|--------|------|-----------------|
| 정확키워드 | `수강신청` | 기본 매칭 |
| 자연어질문 | `수강신청 언제 시작해?` | 문장으로 쳐도 되나 (불용어 처리) |
| 띄어쓰기 변형 | `수강 신청` | 띄어쓰기 달라도 같은 결과 나오나 |
| 오타 | `장학근 신청` | 오타에 견디나 |
| 약어 | `국장`(국가장학금) | 줄임말 알아듣나 |
| 영문 혼용 | `LMS 사용법`, `MJU 뉴스` | 영어 섞여도 되나 |
| 동의어 | `대동제`(축제), `학생식당`(학식) | 다른 말로 쳐도 찾나 |
| 모호 | `요즘 학교 이슈` | 막연한 질문 처리 |
| 노이즈 | `ㅁㅈㄷ`, `ㅎㅇ` | 장난/오입력에 안 터지나 |

→ "정확키워드만 잘 되고 자연어·오타·띄어쓰기엔 약한" 흔한 함정을 정면으로 시험.

### 결과적으로

8개 의도 × 여러 입력형 = **72개 질문**([query-bank.json](query-bank.json)). 각 질문에는
`의도(intent)`, `입력형(form)`, `기대 도메인(expectType)` 꼬리표를 달아, 어느 조합이 약한지
바로 집계할 수 있게 했습니다. 예:

```json
{ "id": "Q20", "query": "성적우수장학금", "intent": "장학", "form": "정확키워드", "note": "복합어" }
{ "id": "Q05", "query": "졸업하려면 학점 얼마나 들어?", "intent": "학사/수강", "form": "자연어질문" }
{ "id": "Q60", "query": "ㅁㅈㄷ", "intent": "엣지", "form": "노이즈", "note": "매칭 실패 기대" }
```

## 5. 어떻게 고쳤나 (Solution)

### 고침 1. 검색어도 저장과 똑같이 형태소로 쪼개기 → "결과 없음" 해결

검색어를 Komoran으로 쪼개 **OR(또는)**로 묶었습니다.

- `"성적우수장학금"` → `성적 | 우수 | 장학금 | 성적우수장학금` (하나만 맞아도 후보로)
- `"수강 신청"` → 공백 뗀 `수강신청`도 추가 → 띄어쓰기 달라도 매칭
- `"ㅁㅈㄷ"` → 쪼갤 의미 단어가 없음 → 검색 안 함(빈 결과)

OR로 묶으니 결과가 잘 나오고(recall↑), 정밀도는 아래 점수로 보정합니다.

### 고침 2. 색인 쓰는 검색만 남기기 → "4초" 해결

```
[전] ... OR similarity(검색용단어들, 검색어) > 0.1   ← 전체 문서 한 줄씩 비교 (느림)
[후] WHERE search_vector @@ to_tsquery(쪼갠검색어)   ← 색인 사용으로 즉시 (빠름)
```

전체를 훑게 만들던 trigram 조건을 빼고, **색인을 타는 전문검색(FTS)만** 남겼습니다.
이게 4초 → 0.2초의 핵심입니다.

### 고침 3. 의미 없는 입력은 DB도 안 친다

`ㅁㅈㄷ` 같은 입력은 쪼갤 단어가 없으므로 **데이터베이스를 아예 조회하지 않고** 빈 결과를 즉시 줍니다.
(예전엔 이런 입력도 4초 걸렸습니다.)

### 고침 4. 제목에 맞는 글을 위로 → "엉뚱한 결과" 해결

점수 계산을 이렇게 바꿨습니다(쉽게 풀면):

```
점수 = (검색어와 얼마나 잘 맞나, ts_rank) × 0.6
     + (제목에 검색어가 하나라도 있으면) +0.25
     + (제목에 검색어 단어가 전부 있으면) +0.40   ← 가장 정확한 글
     + 인기도/카테고리/타입 가중치, 인기검색어 보너스
```

무겁던 trigram 점수 계산은 빼고, **제목 매칭에 가산점**을 줬습니다. 이 계산은 이미 걸러진 소수
문서에만 하므로 느려지지 않습니다.

### 고침 5. 지금 유효한 공지를 위로 / 만료 공지는 아래로 / 학칙은 우대 (이후 추가)

검색 결과에 "이미 끝난" 공지가 위에 뜨는 문제를 점수로 해결했습니다.

```
점수에 추가
     + (마감일이 지난 공지면)  -0.50   ← 후순위로 내림
     + (아직 유효한 공지면)    +0.10   ← 위로 올림
     + (학칙처럼 기한 없는 문서) 만료로 보지 않음 + rule 가중치 0.00 → 0.08
```

- **마감일은 어디서 아나**: 학사일정·학과일정은 일정의 종료일(endDate)을 그대로 쓰고, 공지는
  본문에서 마감 날짜를 추정합니다(`DeadlineExtractor`). 추정 실패 시 "기한 없음"으로 보아 감점하지
  않습니다(틀려도 결과를 망치지 않는 안전 기본값).
- **학칙(rule)** 은 기한 없는 evergreen 문서인데, 예전엔 카테고리 가중치가 0.00(최하위)이라 관련
  검색에서도 묻혔습니다. academic 급(0.08)으로 올려 잘 노출되게 했습니다.
- 이 보정은 `relevance` 정렬에만 적용합니다. `latest`/`oldest`(날짜순)는 날짜 의미를 지키려고
  적용하지 않습니다. 마감 컬럼(`valid_until`)은 점수 계산에만 쓰고 **응답 필드에는 내려보내지
  않으므로 API 스키마는 그대로**입니다(바뀌는 것은 순서뿐).

### 고침 6. 최신 글을 위로 / 뉴스는 아래로 (이후 추가)

학교 정보는 "지금 시점"이 중요합니다. 그래서 관련도가 비슷하면 **최근 글이 위로** 오도록 발행일을
점수에 반영했습니다.

- 공고(공지·학과공지·총학공지)는 최신성을 더 크게(가중치 0.50), 그 외는 0.30.
- 반감기 30일: 한 달 지나면 우대 절반, 두 달이면 1/4, 수개월이면 거의 0.
- 뉴스·방송은 학교 공식 정보보다 우선순위가 낮아 타입 가중치를 음수(-0.05)로 내렸습니다.
  같은 키워드면 공지가 뉴스보다 위로 옵니다.

(역시 `relevance` 정렬에만 적용. `latest`/`oldest` 는 순수 날짜순 유지.)

## 6. 무엇을 어떻게 바꿨나 (변경 구조)

### 처리 흐름 (고친 후)

```
사용자 검색어
  → PgSearchController (GET /api/v2/search/detail)
  → PgUnifiedSearchService (타입/카테고리 정규화, 인기검색어 보너스 준비)
  → UnifiedSearchIndexQueryRepositoryImpl
        1) Komoran으로 검색어 쪼개기 (OR 쿼리 / AND 쿼리)
        2) 의미 단어 없으면 → 빈 결과 즉시 반환
        3) WHERE search_vector @@ (OR쿼리)   ← 색인 사용
        4) 점수 = ts_rank + 제목 가산점 + 가중치 + 유효성 + 학칙 우대 + 최신성(최근↑, 공고 강하게) + 뉴스 하향
  → PostgreSQL (색인 GIN)
  → 결과를 화면용 DTO로 (제목/본문 하이라이트 포함)
```

## 8. 성과 (Results)

- **속도**: 검색 P95 4,201ms → 약 230ms, 평균 약 150ms. (목표 P95 < 500ms 달성)
- **결과없음**: 15/72(21%) → 2/72(3%). 남은 2건은 의미 없는 자모 입력뿐.
- **상위 결과 품질**: `총학생회`→총학 기사, `전자출결`→전자출결 안내, `수강 신청`→수강신청 공지 등
  의도에 맞는 글이 위로. 이전 0건이던 `성적우수장학금`·`출결 규정`·`도서관 좌석`도 정상 노출.
- **유효성 반영(추가)**: 같은 키워드에서 마감 지난 공지가 후순위로 내려가고, 기한 없는 학칙이
  상위로 노출됨을 실 PostgreSQL 통합테스트로 검증(`expired_ranks_below_valid`,
  `evergreen_rule_is_surfaced`). 추가 비용은 인덱스로 걸러진 소수 행에 CASE 한 줄 → 지연 영향 사실상 없음.
- **최신성·뉴스 하향(추가)**: 관련도가 비슷하면 최신 공지가 위로, 뉴스는 공지보다 아래로 정렬됨을
  통합테스트로 검증(`newer_notice_ranks_first`, `notice_outranks_news`). 학교 데이터의 최신성 중요도를 반영.

### 용어 미니 사전

- **recall(재현율)**: "있어야 할 결과가 실제로 나오는 정도". 높을수록 "결과 없음"이 적음.
- **precision(정밀도)**: "나온 결과가 실제로 관련 있는 정도". 높을수록 엉뚱한 게 적음.
- **P95**: 100번 검색 중 95번은 이 시간 안에 끝난다는 뜻(상위 5%만 더 느림). 평균보다 체감에 가까움.
- **GIN 색인**: 단어와 그 단어가 들어 있는 문서 목록을 미리 정리해 둔 색인. 전체 문서 훑기를 피하게 해 준다.
- **trigram**: 단어를 글자 3개씩 겹쳐 자른 조각. 두 단어의 조각이 얼마나 겹치는지로 유사도를 잰다.
- **sargable**: 색인을 활용할 수 있는 검색 조건. 함수로 계산해 비교하는 조건은 비-sargable이라 전체 훑기가 된다.
